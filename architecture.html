ConnectHooks

Architecture & Flow Document

 

Salesforce Managed Package | Namespace: xccch | Version 1.0 | Feb 2026

 

1. What Is ConnectHooks?

 

ConnectHooks is a Salesforce managed package (namespace xccch) that gives external systems forms, apps, or any HTTP client a secure URL to POST JSON data into Salesforce. When the URL receives a payload, ConnectHooks maps the JSON fields to Salesforce object fields and creates records automatically. No custom code is needed from the admin.

The package ships with: a drag-and-drop Mapping UI, a JSONPath engine built entirely in Apex, rate limiting, API key authentication, JSON schema validation, file attachment support, a setup wizard, and a post-install script.

ℹ Think of ConnectHooks as a webhook receiver + field mapper + record creator, all inside Salesforce.

Capability

2. High-Level Architecture

 

The package is split into three layers that work together:

▸ Layer 1 Configuration (Admin does this once)

        Admin opens the ConnectHooks Config LWC tab and runs the setup wizard.

        Wizard creates a public Salesforce Site, assigns permission sets, and creates a File Library.

        Admin then creates Webhooks and uses the Mapping UI to define how JSON maps to Salesforce fields.

        All settings are stored in Custom Settings (ConnectHooks_Config_Settings__c) and Custom Objects.

▸ Layer 2 Runtime (Happens on every webhook call)

        External system sends a POST request to the public site endpoint URL.

        ConnectHooksSubmissionWebService receives the request as a guest user on the public site.

        The service authenticates, validates, processes mappings, creates records, and returns a response.

▸ Layer 3 Support Services (Used by both layers)

        ConnectHooksSOQL all SOQL queries centralised here (runs without sharing for guest access).

        FormResponseUtil shared utilities: key generation, config settings, file creation, type conversion.

        JSONPathEvaluator + JsonParserWithJsonPath custom JSONPath engine built in Apex.

        ErrorMsg + ErrorMessageConfig all error messages defined in one place.

        WebhookLimitTrigger database-level trigger that enforces webhook creation limits.

3. Setup Flow Admin First-Time Configuration

 

When the package is installed, a post-install script runs automatically. After that, the admin completes setup using the ConnectHooks Config tab.

3.1 Package Install Post-Install Script

Class: ConnectHooksPackagePostInstallClass (implements InstallHandler)

Salesforce calls onInstall(context) automatically when the package is installed or upgraded. Two things happen inside this method:

        assignPermissionSet(installingUserId) queries the ConnectHooks_Admin permission set and creates a PermissionSetAssignment record for the installing user so they have full access immediately.

        createDefaultConfigSettings() creates the org-level ConnectHooks_Config_Settings__c custom setting record and sets the default AI backend URL (https://apps.getxccelerance.com/connecthooks) if the field is blank.

ℹ Both operations run WITH SYSTEM_MODE so they succeed even if the installing user lacks certain permissions. Errors are caught and logged they do not break the install.

3.2 Setup Wizard Steps (ConnectHooks Config Tab)

The admin opens the connectHooksConfiguration LWC tab. It shows a 7-step wizard. The steps must be completed in the exact order shown below (matching the UI):

Admin clicks "New Webhook" button. A dialog opens to enter the webhook name. ℹ All settings are stored in ConnectHooks_Config_Settings__c and read back by FormResponseUtil.getConfigCustomSetting() which returns a Map with keys: secureURLCode, siteURL, publicSiteUsed, siteCreationChecked, isJSONStorageChecked, jsonPathApiKey, EndpointUrl, systemAuthKey.

3.3 Creating a Webhook

Class: ConnectHooksConfigurationController.createWebhook(webhookData)

When the admin clicks "Create Webhook" in the UI, this is what happens inside Apex:

1 NAME VALIDATION

Checks webhookData has a name. If blank → error. If contains spaces → error (ErrorMsg.webhookNameNoSpaces).

 

2 DUPLICATE CHECK

Queries CH_Webhook__c by name. If found → error (webhook already exists).

3 GENERATE SECURE CODE

Calls FormResponseUtil.createSecureURLCode() 48-char random lowercase/number string with dashes.

4 GENERATE API KEY

Calls FormResponseUtil.createAPIKeyTokenString() 48-char random alphanumeric (upper + lower + digits).

5 INSERT RECORD

Inserts CH_Webhook__c with: Webook_Name__c, Secure_URL_Code__c, Enc_Api_Key__c, is_Active__c = true.

6 TRIGGER FIRES

WebhookLimitTrigger → WebhookLimitTriggerHandler.handleBeforeInsert() checks if user has exceeded the creation limit from CH_Config_Meta_Store__mdt (Free_Edition.Webhook_Creation_Limit__c). Counts ALL webhooks ever created by user (including deleted). If limit exceeded → addError() blocks insert.

ℹ The endpoint URL format is: {siteSecureURL}/services/apexrest/xccch/ConnectHooksSubmissionService/{webhookName}/{secureCode}

3.4 Setting Up Field Mappings

Class: MappingController.saveMappingData(mappingRecords, jsonSchema, hierarchicalObjects)

The admin uses the Mapping UI (mappingUi LWC) to define how JSON fields map to Salesforce fields. Here is the exact flow when the admin clicks Save:

1 SCHEMA SAVE

If a JSON schema was uploaded, saveMappingData upserts a JSON_Schema__c record linked to the webhook. This schema is used later to validate incoming requests.

2 HIERARCHY FLATTEN

flattenHierarchyWithIds() recursively walks the object tree from the UI (parent → child objects) and builds two maps: objectDataMap (objectId → name) and parentChildRelationships (childObjectId → parentObjectId).

3 UPSERT MAPPING OBJECTS

For each object in the tree, creates or updates a CH_Mapping_Object__c record with: Custom_Object__c (target Salesforce object API name), Webhook__c (link to webhook), Parent_Mapping_Object__c (link to parent object for hierarchy).

4 DELETE ORPHANS

Any CH_Mapping_Object__c records that existed before but are not in the new save are deleted along with their child CH_Mapping__c records.

5 INSERT FIELD MAPPINGS

For each field line in mappingRecords, inserts a CH_Mapping__c record with: Fields__c (Salesforce field API name) and JsonPath_Expression__c (e.g. $.orders[*].id).

ℹ ObjectFieldController.getSObjects() provides the object list to the UI. getObjectFields() provides fields. getRequiredFields() shows which fields are mandatory. JSONPathAPIController evaluates JSONPath expressions live in the UI for field preview.

 

4. Runtime Flow What Happens on Every Webhook Call

 

This is the core of ConnectHooks. When an external system sends JSON to the endpoint URL, the following 9-step process runs inside ConnectHooksSubmissionWebService.handleWebhookRequest().

ℹ The class is declared: @RestResource(urlMapping="/ConnectHooksSubmissionService/*") global with sharing. It runs as the site guest user on the public Salesforce site.

Step 1 Extract Webhook Identity from URL

Method: handleWebhookRequest()

The incoming URL looks like: .../ConnectHooksSubmissionService/MyWebhook/a1b2c3d4-e5f6g7h8-...

String urlPath = req.requestURI.substringAfter('/ConnectHooksSubmissionService/');

List<String> pathParts = urlPath.split('/');

webhookName = pathParts[0]; // e.g. MyWebhook

secureCode = pathParts[1]; // e.g. a1b2c3d4-e5f6g7h8-...

If the URL does not have at least 2 parts → returns HTTP 400 with InvalidUrlFormat error.

Step 2 Parse the JSON Body

Method: handleWebhookRequest()

String jsonBody = req.requestBody.toString();

Map<String,Object> jsonData = (Map<String,Object>) JSON.deserializeUntyped(jsonBody);

Blank body → HTTP 400 EmptyRequestBody. The full JSON map is kept in memory for all subsequent steps.

Step 3 Extract the API Key

Method: extractAPIKey(req, jsonData)

The method checks three places in order:

        Priority 1: req.headers.get("x-api-key") the HTTP header

        Priority 2: jsonData.get("api_key") a root-level key in the JSON body

        Priority 3: jsonData.get("x-api-key") alternate root-level key in the JSON body

If no key found anywhere → HTTP 401 MissingApiKey error.

Step 4 Find and Validate the Webhook Record

Method: findAndValidateWebhook(webhookName, secureCode, apiKey)

// Calls ConnectHooksSOQL.getWebhookConfig(webhookName) -- runs WITHOUT SHARING

// Returns CH_Webhook__c WHERE Webook_Name__c = :name AND is_Active__c = TRUE

Three checks must all pass:

        Webhook record must exist and be active → else HTTP 404

        Secure code in URL must match Secure_URL_Code__c field → else HTTP 401 InvalidSecureCode

        API key must match Enc_Api_Key__c via EncryptionUtil.validateApiKey() → else HTTP 401 InvalidApiKey

ℹ EncryptionUtil.validateApiKey() does a direct string equals. Salesforce EncryptedText fields encrypt at the platform level, so the value returned by a non-USER_MODE query is already the plain-text key for comparison.

 

▸ 4a. Check Rate Limit

Method: checkAndUpdateRateLimit(webhook, res)

This method has 5 internal parts:

        Part 1: Read API_Rate_Limit__c and Time_Window_Hours__c from CH_Config_Meta_Store__mdt (Free_Edition record).

        Part 2: Get ConnectHooks_Rate_Limit_Tracker__c.getInstance(webhook.Id) a custom setting record keyed by webhook ID.

        Part 3: If the time window has expired (hours since Last_Reset_Time__c >= Time_Window_Hours__c), reset Total_API_Hits__c to 0.

        Part 4: If Total_API_Hits__c >= rateLimit → throw WebhookException (HTTP 429 RateLimitExceeded).

        Part 5: Increment Total_API_Hits__c by 1 and upsert the tracker record.

ℹ If the rate limit metadata is missing, the system logs a warning but does NOT block the request it only blocks if a valid config exists and the limit is hit.

Step 5 JSON Schema Validation (Optional)

Method: validateJSONSchema(jsonData, schemaJson)

Only runs if the webhook has a JSON schema stored in CH_Webhook__c.Schema__c or JSON_Schema__c.

Map<String,Object> schema = (Map<String,Object>) JSON.deserializeUntyped(schemaJson);

List<Object> required = (List<Object>) schema.get("required");

// Checks each required field name exists as a key in jsonData

If any required field is missing → HTTP 400 JsonSchemaValidationFailed. If no schema is configured, this step is skipped entirely.

Step 6 Check for Custom Handler (Optional Bypass)

Class: ConnectHooksSOQL.getCustomHandlerConfig(webhook.Id)

Checks if a ConnectHooks_Custom_Handler__c custom setting record exists for this webhook and its Is_Active__c is true. If yes, the standard mapping flow is bypassed entirely and executeCustomHandler() runs instead.

Method: executeCustomHandler(className, jsonPayload, res)

        Type.forName(className) loads the Apex class dynamically (tries without namespace first, then with "xccch").

        Verifies the class implements ICHCustomHandler interface (instanceof check).

        Calls customHandler.execute(jsonPayload) the developer writes all logic inside this method.

        Returns HTTP 200 with a success message. The custom handler is responsible for any DML.

ℹ ICHCustomHandler is a simple interface with one method: void execute(String jsonPayload). This lets any developer write their own processing logic without modifying the core package.

Step 7 Load Mapping Configuration

Method: ConnectHooksSOQL.getMappingsForWebhook(webhookId) runs WITHOUT SHARING

SELECT Id, Name, xccch__Custom_Object__c, xccch__Parent_Mapping_Object__c,

(SELECT Id, xccch__Fields__c, xccch__JsonPath_Expression__c FROM CH_Mappings__r)

FROM CH_Mapping_Object__c WHERE xccch__Webhook__c = :webhookId

Returns all CH_Mapping_Object__c records with their child CH_Mapping__c field mappings. If empty → HTTP 400 NoMappingsConfigured.

Step 8 Process Mappings and Create Records

Method: processMappings(mappingObjects, jsonData, webhook) this is the mapping engine

For each CH_Mapping_Object__c, the engine does:

▸ 8a. Detect Collection vs Single

isCollectionMapping() checks the JSONPath of every field mapping. If any path contains [*], [:], [?(, or .. → it is a collection mapping (creates multiple records). Otherwise it is a single-record mapping.

▸ 8b. Collection Path Extract Multiple Records

Method: extractMultipleRecords(fieldMappings, jsonData)

Uses JSONPathEvaluator.evaluatePath() with the wildcard path to get all array items. For each item it builds a field-value Map. The internal key __PARENT_INDEX__ is stored to track which parent record a child record belongs to.

▸ 8c. Single Path Extract One Record

Method: extractFieldValues(fieldMappings, jsonData)

Calls JSONPathEvaluator.evaluatePath() for each field mapping. Converts the raw value to the correct Apex type using FormResponseUtil.getAdjustedDataTypePerField() which reads Schema.DisplayType for each field.

▸ 8d. Resolve Parent Relationship

Method: resolveParentRelationshipByMappingId(fieldValues, mappingObj, ...)

If the mapping object has a Parent_Mapping_Object__c set, the engine looks up the already-created parent record ID from createdRecordsByMappingId map. Then findRelationshipField() scans all REFERENCE fields on the child object using Schema.getGlobalDescribe() to find which field points to the parent type. That field is automatically added to fieldValues.

ℹ This is how parent-child records are automatically wired e.g. if you map Order → OrderItem, the OrderId field on OrderItem is set automatically without any extra configuration.

▸ 8e. Create the Record Dynamically

Method: createDynamicRecord(objectApiName, fieldValues)

SObject record = Schema.getGlobalDescribe().get(objectApiName).newSObject();

// Sets each field after type conversion

Database.upsert(record, externalIdField, true); // via ConnectHooksSOQL.safeUpsert()

If an external ID field is detected, it upserts. Otherwise it inserts via ConnectHooksSOQL.safeInsert().

Step 8 (cont.) JSON Storage

If webhook.Enable_Storage_Of_Form_Submission_JSON__c is true, storeFormSubmission() creates a Form_Submission__c record with the raw JSON body, the webhook name, and the first created record ID. This gives admins a full audit trail.

Step 9 Return Response

If all mappings succeeded → HTTP 200. If some failed → HTTP 207 Multi-Status. Errors and created record IDs are included in the JSON response body.

{ "success": true, "webhook": "MyWebhook", "recordsCreated": 3,

"recordIds": ["001...", "003...", "a01..."], "errors": [] }

5. JSONPath Engine How JSON Values Are Extracted

 

ConnectHooks does NOT use any external library for JSONPath. The entire engine is built in Apex across two classes: JsonParserWithJsonPath (tokenizer and parser) and JSONPathEvaluator (expression evaluator).

5.1 JSONPathEvaluator

Main method: evaluatePath(Object parsedJson, String jsonPath)

Supports the full RFC 9535 JSONPath syntax:

Expression

Example

What It Returns

Root

$.name

Value of "name" key at root

Dot notation

$.user.email

Nested key access

Bracket notation

$["user"]["email"]

Same as dot but with brackets

Wildcard

$.orders[*].id

All "id" values from orders array

Array index

$.orders[0].id

First order id

Array slice

$.orders[0:3]

First 3 orders

Recursive descent

$..email

All "email" keys at any depth

Filter expression

$.orders[?(@.status=="active")].id

IDs of active orders

Multiple paths

evaluatePaths(json, paths)

Map of path → value for a list of paths

 

ℹ evaluatePaths() is used by the Mapping UI live preview to evaluate multiple JSONPath expressions at once against a sample JSON payload.

5.2 JSONPathAPIController

REST endpoint: @RestResource(urlMapping="/api/jsonpath/*")

POST: Accepts { jsonData, jsonPaths[] } or { jsonData, jsonPath, operation: "single" }. Calls JSONPathEvaluator and returns results. Supports an optional includeDebugInfo flag.

GET (health check): Returns { status:"OK", service:"JSONPath API", version:"1.0" }.

ℹ This endpoint is used internally by the Mapping UI to evaluate JSONPath expressions as the admin drags fields. It is also used by the JSONPath Generator LWC to call the AI backend.

8. Security Design

 

Control

How It Works

API Key (first factor)

x-api-key sent by caller. Compared against EncryptedText field. Invalid key → HTTP 401.

Secure URL Code (2nd factor)

48-char code in the URL itself. Must match exactly. Invalid → HTTP 401.

Rate Limiting

Per-webhook counter in Custom Setting. Exceeding limit → HTTP 429. Resets after time window.

Webhook Creation Limit

Database trigger (before insert) prevents users creating more than the allowed number of webhooks cannot be bypassed via API.

USER_MODE Queries

All controller-layer SOQL uses WITH USER_MODE or AccessLevel.USER_MODE to enforce FLS and object permissions.

SYSTEM_MODE for Files

ContentVersion insert uses WITHOUT SHARING inner class (WithoutSharingContext) because the site guest user cannot own Files. This is an intentional, documented exception.

Guest User Permissions

ConnectHooks_Public_Site_User_Permissions gives only the minimum access: read on mapping objects + access to the REST Apex class. No write permissions granted.

Schema Validation

Optional. Checks required fields in JSON before any processing. Protects against incomplete payloads.

No Hardcoded Secrets

All keys and URLs stored in Custom Settings and Custom Metadata never in Apex code.

Custom Handler Safety

ICHCustomHandler implementors must pass an instanceof check before execute() is called.

9.3 Mapping UI Validation Rules

Two rules are enforced client-side in mappingDropzone.js before a mapping is accepted:

▸ Rule 1 Type Compatibility (typeMappingRestrictions.js)

JSON Type

Blocked Salesforce Field Types

string

INTEGER, DOUBLE, LONG, DECIMAL, CURRENCY, PERCENT, BOOLEAN, CHECKBOX, LOCATION, GEOLOCATION

number

EMAIL, PHONE, URL, MULTIPICKLIST, BOOLEAN, CHECKBOX, DATE, TIME, DATETIME

boolean

EMAIL, PHONE, URL, PICKLIST, MULTIPICKLIST, all numeric types, DATE, DATETIME, TIME, LOCATION, GEOLOCATION

null

All types null cannot be mapped to any field

 

▸ Rule 2 Array-Bounded Object Restriction

All fields from the same JSON array object (e.g. $.orders[*]) must map to the same Salesforce object. This ensures one record is created per array element with all its fields. Mixing $.orders[*].id → Account with $.orders[*].name → Contact is blocked.

12. Conclusion

 

ConnectHooks is a complete, self-contained Salesforce managed package that solves a very specific problem: how do you get data from an external form, app, or service into Salesforce records automatically without writing custom code for every integration?

The answer ConnectHooks provides is simple from the admin side: you set up a webhook once, draw lines between JSON fields and Salesforce fields in a visual drag-and-drop UI, and from that point forward every form submission automatically becomes a Salesforce record.

What Makes ConnectHooks Work Well

There are four design decisions that make the project work together cleanly:

1 EVERYTHING GOES THROUGH ONE REST CLASS

ConnectHooksSubmissionWebService is the single entry point. All authentication, validation, mapping, and record creation happen in sequence inside one handleWebhookRequest() call. This makes the flow easy to debug and audit.

2 MAPPING IS STORED AS DATA, NOT CODE

The field mapping rules live in CH_Mapping__c and CH_Mapping_Object__c records. This means changing a mapping never requires a code deployment the admin just edits records in the UI. The runtime service reads these records fresh on every call.

3 THE JSONPATH ENGINE IS BUILT ENTIRELY IN APEX

JsonParserWithJsonPath and JSONPathEvaluator handle all JSON traversal without any external libraries. This was necessary because Salesforce managed packages cannot include npm packages or external JS. The engine supports the full RFC 9535 JSONPath syntax including wildcards, filters, and recursive descent.



4 SECURITY IS LAYERED, NOT SINGLE-POINT

Every incoming request must pass four independent checks: the URL must contain the correct secure code, the API key must match the encrypted value, the webhook must be active, and the rate limit must not be exceeded. Additionally, a database trigger ensures webhook limits cannot be bypassed through any interface.

ℹ For advanced use cases, developers can implement the ICHCustomHandler interface in their own Apex class and register it against a webhook. ConnectHooks will then call their class instead of the standard mapping engine, giving full programmatic control while still handling authentication and rate limiting automatically.
